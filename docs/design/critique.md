# Burnfront v1 — Completeness & Risk Review

## P0 — Contradictions between the three docs (must be resolved BEFORE the WS-01 contract freeze; freezing specs while these are open freezes bugs)

1. **Daily day boundary is defined three different ways.** Architecture: UTC. Product §7: "dailies flip at local midnight (Wordle convention)". Orchestration WS-07: "Europe/Brussels day boundary." This decides streak logic, the 48h batch window, `puzzle_fetches` credit, the streak-protection email, and offline pre-cache. **Fix:** one ADR, day one — recommend UTC content boundary + local-midnight *display* flip is the worst of both; pick pure UTC (architecture's version) and document it in-app; kill "local midnight" from product copy. → WS-01, ADR-0002.
2. **Three incompatible auth designs.** Architecture: passwordless magic link + Google + Apple. Product: magic link only, "no social providers v1". Orchestration WS-06: Fortify with **password reset** (passwords!). **Fix:** magic link only at v1 (smallest GDPR/abuse surface, no OAuth app registrations, no $99 Apple dev account for a web launch); keep `auth_identities` table so providers are additive. Rewrite WS-06 brief — Fortify password flows are dead weight. → WS-01/WS-06.
3. **Two different repo layouts and file names.** `api/` vs `services/api`; `contracts/` vs `specs/`; `packages/api-client` (generated, CI-verified) vs WS-09 "client.ts generated, committed"; `packages/ui-web` exists only in orchestration; `content.burnfront.com` vs `cdn.burnfront.com`; `/d/{date}` vs `/daily/{date}`. **Fix:** one canonical tree in WS-00's CLAUDE.md; agents building from conflicting docs will fork the repo structure in week 1. → WS-00.
4. **Two incompatible DB schemas.** Orchestration's 8-table `db-schema.sql` is missing `puzzle_fetches` (the anti-cheat time anchor — WS-07's validator can't work without it), `auth_identities`, `board_ratings`, `content_imports`, `rating_events` (has `rating_history` instead), and adds an `entitlements` table the architecture explicitly rejected in favor of `plan`/`pro_until` columns. **Fix:** architecture §3 schema wins; regenerate `specs/db-schema.sql` from it. → WS-01.
5. **Endless rated or not.** Architecture: "endless is unrated in v1" (locked, with good reasons). Product §5: rating delta "after every rated solve (daily + endless)", board rating shown on endless boards, "rating exits calibration ~10 solves if they touch endless". WS-11 punts to RATING.md. With daily-only rating, Glicko-2 gets **one game per day** — "calibrating 6/10" takes 10 days and the number-goes-up loop dies. **Fix:** decide now; recommended: endless rated at reduced weight *server-side using the submitted board spec* — server re-runs the 30-line BFS on `endless_spec` (validity is checkable; only uniqueness isn't, and uniqueness doesn't matter for rating a solve). Update RATING.md, product §5, architecture §3 together. → WS-01/RATING.md.
6. **Leaderboards contradict "no public profiles".** Architecture ships `GET /leaderboards/daily/{date}` and time-ordered ranks; product says no handles, no public profiles, percentile-only stats card. Named leaderboards without moderation, handle-abuse policy, or robust anti-cheat is a liability. **Fix:** CUT the leaderboard endpoint from v1; keep percentile + rank-number only (computable from aggregates, no names exposed). → WS-01/WS-07.
7. **Analytics stack contradiction.** Product: "self-hosted Plausible + custom events". WS-19: first-party events table, no third parties. Self-hosted Plausible drags ClickHouse onto a CPX31 — absurd for v1. **Fix:** CUT Plausible; first-party `events` table + weekly digest (WS-19) is the whole stack. Add pageview-level `first_seen`/session events to WS-19's list so the D1/D7 funnel is actually computable. → WS-19.
8. **JS budget contradiction.** Product: deferred JS ≤ 90KB gz total (engine ≤ 30KB). Orchestration gate 7: initial JS ≤ 200KB gz. React 19 + TanStack Router cannot meet 90KB. **Fix:** two budgets: landing page (Blade + inlined hero, ≤ 90KB deferred) and SPA (≤ 200KB initial); write both into `lighthouserc.json`. → WS-01/WS-17.
9. **Hetzner box disagreement:** CPX31 (arch) vs CX22 (WS-16) — CX22 (2 vCPU/4GB) cannot run nginx+fpm+Redis+PG+Horizon+staging comfortably. **Fix:** CPX31; fix WS-16 brief. → WS-16.
10. **Academy lesson count:** 7 lessons (product, mapped 1:1 to toolkit) vs 6 (WS-12) vs "five deduction arguments". **Fix:** 7, per product §5; fix WS-12 brief and the academy pack spec. → WS-12/WS-05.

## P1 — Missing entirely (each needs a brief; most are small)

11. **The anonymous→account merge has no endpoint, no schema, no workstream, and a gaping abuse hole.** Product depends on it (the day-3 nudge is "honest" only if merge works). Server re-validates correctness of uploaded local solves but **cannot validate when they were solved** — anyone can fabricate a 100-day streak at signup by replaying archive boards. **Fix:** new endpoint `POST /me/import` (idempotent, ≤ N items); merged streak credit capped at, say, 7 days and only for dates whose boards were published before the claimed solve date; merged solves flagged `imported=true`, leaderboard/percentile-ineligible; rating seeded from merged solves at high RD. New WS-20 (api + web), ~1 session each side. → new WS-20.
12. **Streak freeze exists only as product copy.** No schema columns (`streaks` has none), no rollover job, no workstream, and it interacts with the streak email ("has 4 hours" vs auto-freeze). **Fix:** add `freeze_available_at`/`frozen_dates` to `streaks`, a scheduled `streaks:rollover` command applying freezes at day boundary, and fold into WS-07 brief. → WS-07/WS-01 schema.
13. **Playtesting is absent from the build schedule.** Product §8 mandates 2 waves (~3 weeks) with a go/no-go gate; orchestration goes P4 hardening (wk 7–8) → P5 launch (wk 9) with zero playtest window. **Fix:** insert P4.5 "playtest soak" (2–3 weeks, staging + wave-2 instrumentation) between P4 and P5; realistic launch ≈ week 11–12. → orchestration §7.
14. **The streak-protection email needs a timezone the schema doesn't store.** "20:00-local" requires `users.timezone`; nothing captures it. Also no workstream implements this email, double opt-in, or the send-time scheduler. **Fix:** add `users.timezone` (set client-side at signup), a `notifications:streak-risk` scheduled command, and put it in a workstream (extend WS-18 or new WS-21 "transactional email": magic link, deletion confirm, export link, streak alert — templates, EU-region ESP config, list-unsubscribe header). → new WS-21.
15. **Frontend error tracking is missing.** Nightwatch sees PHP only; a broken SPA release is invisible until a human notices. **Fix:** first-party `POST /api/v1/errors` beacon (rate-limited, sampled, no third parties — consistent with the CSP), surfaced in the weekly digest + Nightwatch log channel. → WS-19.
16. **Broken-published-daily runbook missing.** Pipeline guarantees are pre-publication; nothing covers a daily discovered broken *live* (bad OG image, grader mislabeled, engine-parity bug making valid solves rejected). Also: calendar "rebuilt from the re-sorted pool" (product §8) contradicts "immutable per content_version" and breaks clients that pre-cached tomorrow's board. **Fix:** rule: published dates are immutable T-48h out, calibration re-sorting applies to future dates only; runbook section "pull a daily" = publish replacement content_version + streak amnesty flag for that date + in-app notice. → WS-05/WS-18 RUNBOOK.
17. **CDN-down fallback missing.** Daily JSON lives only on R2; if `content.burnfront.com` is down at 00:00 UTC the product is dead and only the freshness alert (which checks *after* the flip, WS-18) notices. **Fix:** freshness check at T-2h for tomorrow's board; `GET /daily/{date}` gains an origin-served `puzzle` fallback body behind a flag; client falls back to API when CDN fetch fails. → WS-07/WS-18.
18. **Magic-link security is unspecified.** Token TTL, single-use, hashed at rest, constant-response (no account enumeration), per-*email* throttle (5/min/IP doesn't stop distributed email-bombing of one victim), session rotation on consume. **Fix:** spec these in openapi.yaml + WS-06 acceptance criteria; add `POST /auth/magic-link` per-email throttle 3/hour. → WS-06.
19. **No security-review workstream.** Gates cover code quality; nobody runs a threat pass over auth, solve submission, import/merge, the events endpoint, or the export-link (signed URL? expiry?). **Fix:** add a hardening session in P4 running `/security-review` against `services/api` + the batch/import/events endpoints; define CSP + security headers (HSTS, frame-ancestors, CSP with hash'd inline for the landing hero) as a deliverable — currently CSP is *assumed* by product ("CSP-friendly") and *tested* by WS-14 but never *authored*. → new hardening session, P4.
20. **GDPR paperwork beyond export/delete.** Missing: processor inventory + DPAs (Hetzner ✓EU, Cloudflare (US, DPF — must be in the privacy policy), ESP — note **Postmark/Resend are US**; product's "EU-region ESP" contradicts architecture's Postmark/Resend, pick an EU region or EU processor and document SCCs/DPF), Art. 30 record of processing (a one-page markdown suffices for a solo controller), retention schedule (see #21), breach-notification note in RUNBOOK, and a children/age statement in terms (puzzle games attract minors; Belgian GDPR consent age is 13). **Fix:** `docs/gdpr.md` + privacy-policy checklist as WS-14 inputs; owner signs off. → WS-14/owner.
21. **No data-retention policy — replay logs are the problem child.** Storing full per-solve replay event logs + ip_hash/ua_hash **indefinitely** for hypothetical future anomaly detection strains GDPR purpose-limitation/minimization. **Fix:** retention rules in schema + scheduled purge: replays 90 days (unless flagged `suspect`), ip/ua hashes 90 days, events table aggregated then row-purged at 13 months; state all of it in the privacy policy. → WS-06/WS-19.
22. **Erasure semantics undefined.** `DELETE /me` "queued job" — but do solves/rating_events get deleted or anonymized? Board ratings and daily stats are derived from them. **Fix:** anonymize (null user_id, keep aggregates) — document in privacy policy; test in WS-14's delete e2e (which currently asserts "removes rows" — wrong if anonymizing). → WS-06/WS-14.
23. **i18n never addressed.** Dutch-speaking owner, EU audience. **Fix:** launch EN-only (right call — the fiction/voice is English-native), but make COPY.md keyed strings loaded through one message module from day one so NL is a translation file, not a refactor; use `Intl` for dates/numbers. One paragraph in WS-01's COPY.md spec. → WS-01/WS-09.
24. **Social-proof counter has no data source.** "12,408 crews contained" — anonymous solves never reach the server, so the counter counts only account-holders and will read embarrassingly low for months. **Fix:** count anonymous *completions* via the first-party events endpoint (`solve_complete`, no account needed) and show that; label it "crews", dedupe by anon id, accept fuzz. → WS-19/WS-15.
25. **`official_ms` clamp is one-directional.** `min(client_ms, server_window)` accepts any *understated* client_ms above the perceptual floor (8 breaks → 2s floor; claim 15s, top the daily percentile). **Fix (v1-cheap):** also require `client_ms ≥ replay duration` and `≥ fetch→submit − idle allowance`; below → `suspect`. Full replay audit stays phase-2, but this closes the dumb lie. → WS-07.
26. **SEO odds and ends:** robots.txt, `noindex` on `/play|/me|/settings`, custom 404 (future daily dates), apex/www canonical redirect, `sitemap.xml` including dated dailies for the past N days. **Fix:** checklist in WS-15 acceptance. → WS-15.
27. **Accessibility gaps in an otherwise decent story:** (a) 400/800ms **hold-to-reveal** hint buttons violate pointer-cancellation/motor-accessibility norms — provide a plain double-activation alternative; (b) no declared conformance target — state WCAG 2.1 AA in specs; (c) SPA route-change focus management + `aria-live` for burn replay/CONTAINED; (d) drag-paint has keyboard parity (good) — add it to WS-04 acceptance explicitly, not just "announcements". → WS-04/WS-09/WS-13.
28. **Backup design is slightly wrong.** "Hourly WAL-less incremental via pg_dump of hot tables" — pg_dump is neither incremental nor cross-table-consistent when scoped to tables. **Fix:** pgBackRest or wal-g → R2 (EU-jurisdiction bucket, encrypted): nightly full + continuous WAL = minutes-RPO for less effort; keep the quarterly restore drill; state RTO/RPO (e.g., 4h/15min) in RUNBOOK. → WS-16/WS-18.
29. **Legal pages have no authoring task.** `/privacy`, `/terms`, `/imprint` copy is "owner-supplied" but nothing schedules drafting them (Belgian imprint: name, address, email, VAT/BCE number if applicable; governing law). **Fix:** owner task in P3 with an agent-drafted first pass reviewed by the owner (and ideally a human lawyer skim before launch). → owner/WS-14.
30. **Cookie/ePrivacy claim needs one nuance.** "No banner needed" holds for strictly-necessary localStorage, but the persistent **anonymous analytics id in localStorage used for D1/D7 cohorts** is not strictly necessary under ePrivacy — Belgian DPA (APD/GBA) reads this strictly. **Fix:** either accept low risk and disclose clearly in the privacy policy (common Plausible-style posture), or cohort on a 24h-rotating id. Decide consciously, write it down. → WS-19/privacy policy.

## P2 — Corrections / hygiene

31. **Staging on the prod box** shares PHP-FPM/Redis/PG with prod and auto-deploys `main` — a bad staging migration can eat prod IO. Acceptable for v1, but isolate: separate PG cluster/port + FPM pool memory caps; note in RUNBOOK. → WS-16.
32. **Content rollback path:** `content:import` has no `content:rollback {version}`; add it (repoint `daily_puzzles` rows to prior version in one transaction). → WS-05/WS-07.
33. **Export link security:** `GET /me/export` "emailed link" — spec signed URL, 24h expiry, single download, re-auth required. → WS-06.
34. **Events endpoint abuse:** WS-19 needs rate limit (e.g., 60/min/session), payload schema validation, and batch caps or it's a free write-amplification vector into Postgres. → WS-19.
35. **`/about` page** (the provably-fair/press-kit page, product §1) is in no workstream — fold into WS-15. → WS-15.
36. **Vector count disagreement:** architecture wants ~500 adversarial burn cases; orchestration freezes 50 seeds. 50 is too thin for the crown jewels. Use architecture's numbers. → WS-01.
37. **Client-side Ed25519 verification (WS-10)** is near-pointless (JS ships from the same trust domain that pins the key) — keep server-side import verification, make client verification optional/dropped. Minor CUT. → WS-10.
38. **`GET /daily/{date}` vs `/daily/{date}/stats`** endpoint drift between docs — reconcile in openapi.yaml (one endpoint with stats embedded is fine at this scale). → WS-01.
39. **Redis queue durability:** Horizon jobs (rating, exports) vanish on Redis restart with default persistence — enable AOF or accept + document (ratings recomputable from `rating_events`, exports re-requestable). → WS-16.
40. **`handle` column exists but product has no public profiles and no handle moderation** — keep the column, don't expose handles anywhere in v1, add a reserved-words check when it ships. → WS-06.

## CUT from v1 (over-engineering)

41. **Sign in with Apple + Google** — see #2; magic link only. Saves OAuth registration, Apple dev account, account-linking QA. → WS-06.
42. **Leaderboard endpoint + Redis-sorted-set plan** — see #6. Percentiles only. → WS-07.
43. **Self-hosted Plausible** — see #7. → WS-19.
44. **Monthly partitioning of `solves` from day one** — at ~2k rows/day this is years of headroom; partitions complicate every FK/index/ORM interaction and the unique constraints the design relies on. A plain table + the documented 50k-DAU migration trigger is enough. (If kept, at minimum prove the two partial unique indexes work under partitioning — they constrain per-partition, not globally, unless the partition key is in the index.) → WS-06.
45. **`POST /solves/batch` offline sync** — v1 PWA needs "replay a solved daily offline", not offline *submission* with 48h-late reconciliation, which drags in the hardest streak edge cases. Cut; submit-on-reconnect for the single in-progress daily is enough. → WS-07/WS-10.
46. **Nightly `ratings:recalibrate` live board-rating re-fit (WS-08)** — with daily-only rated solves there's barely data; seed board ratings from pipeline grade and revisit post-launch. Keep `rating_events` capture. → WS-08.
47. **Pixel-diff acceptance ≤ 2% vs prototype (WS-04)** — screenshot-diffing a rewrite against a prototype guarantees flaky red CI; replace with fixture-based DOM assertions + one human look. → WS-04.
48. **90 days of dailies committed as fixture (WS-05)** contradicts "artifacts are build outputs, NOT committed" — commit ~7 days as test fixtures, generate the rest in CI/release. → WS-05.

## Meta-risk

49. **The three documents were clearly written independently and disagree on ~15 load-bearing details** (#1–#10, #36, #38, #44, #45, #48). The orchestration plan freezes contracts in WS-01 sessions 2–3; if agents transcribe from whichever doc they were handed, the contradictions get frozen. **Fix:** before Session 2, the lead produces a single one-page `docs/decisions.md` resolving every item in P0 above, owner-approved; WS-01 authors contracts from that page only, and the three design docs are demoted to background reading.
